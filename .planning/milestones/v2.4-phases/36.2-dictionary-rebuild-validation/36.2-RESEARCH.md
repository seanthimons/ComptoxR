# Phase 36.2: Dictionary Rebuild Validation - Research

**Researched:** 2026-04-27
**Phase:** 36.2 - Dictionary Rebuild Validation
**Goal:** Plan semantic adjudication for regenerated source-backed ECOTOX lifestage artifacts.

## Research Question

What do we need to know to plan Phase 36.2 well?

The current implementation already has deterministic routing, route-family ontology priorities, full ECOTOX taxon intersections, curated exception reporting, explicit unresolved derivation placeholders, and a Phase 36.2 validation script. The remaining planning risk is not missing ontology coverage. The risk is semantic overconfidence: a resolved source concept can be real and source-backed while its global `harmonized_life_stage` or `reproductive_stage` derivation is wrong for a specific ECOTOX term/context.

## Current State

Relevant committed or working artifacts:

- `inst/extdata/ecotox/lifestage_baseline.csv` has 139 ECOTOX lifestage rows with source-backed provider metadata.
- `inst/extdata/ecotox/lifestage_derivation.csv` maps `source_ontology + source_term_id` to derived harmonized fields.
- `inst/extdata/ecotox/lifestage_curated_exceptions.csv` documents retained curated rows and allowed reason families: `semantic_change`, `ambiguous_route`, `no_source_backed_candidate`.
- `inst/extdata/ecotox/lifestage_taxon_intersections.csv` provides every observed ECOTOX term/taxon route intersection, including the reported species-group context (`eco_group`, `kingdom`, `class_name`), `route_family`, signal counts, and dominant route flags.
- `inst/extdata/ecotox/lifestage_route_ontology_priorities.csv` externalizes route-family ontology priority.
- `dev/lifestage/validate_36.2.R` currently validates schemas, alias coverage, unresolved derivation placeholders, orphan derivation rows, cross-check coverage, and GO contamination.
- `tests/testthat/test-eco_lifestage_data.R` already locks route policy, curated exceptions, explicit unresolved derivation rows, and several high-risk policy decisions. Existing Sapling checks should be treated as example coverage, not as the model for a permanent one-row rule.

Important runtime helpers:

- `.eco_lifestage_taxon_route_lookup()` prefers the full taxon intersection artifact before falling back to dev-only dominant profile data.
- `.eco_lifestage_rank_candidates()` ranks by route-family ontology priority, candidate score, source ontology, and source term ID.
- `.eco_lifestage_derive_fields()` still joins resolved rows to a source-ID-level derivation map.
- `.eco_lifestage_derivation_coverage_report()` checks derivation key coverage but does not prove semantic defensibility by ECOTOX term/context.

## Key Planning Findings

1. The source-ID-level derivation map is intentionally simple, but Phase 36.2 must test whether it is too coarse.
2. A semantic adjudication artifact should be data, not hidden code. It needs to be diffable, reviewable, and enforceable by `testthat` and `dev/lifestage/validate_36.2.R`.
3. The artifact should join baseline rows, derivation rows, taxon route context, curated exceptions, and candidate replacement metadata. This gives reviewers enough evidence to decide whether the derived fields are defensible.
4. The artifact should not automatically promote replacements. The safe action set is explicit: keep current derivation, retain curated exception, policy-hold unresolved, or flag for context-aware derivation.
5. `Sapling` is a useful example, not an iron-clad regression rule. It illustrates the broader invariant: any replacement that changes `harmonized_life_stage` or `reproductive_stage` needs explicit adjudication.
6. Additional ontologies should only be considered after the artifact shows `no_source_backed_candidate` for a specific ECOTOX term/context.
7. The current runtime resolver chooses candidates from an `org_lifestage` input and uses the dominant taxon route as ranking context. Phase 36.2 must compensate by preserving all observed `org_lifestage + eco_group + kingdom + class_name` contexts in adjudication and by flagging lifestages whose non-dominant contexts could change semantics.

## Recommended Artifact Contract

Create `inst/extdata/ecotox/lifestage_semantic_adjudication.csv` with one row per observed ECOTOX lifestage/species-group context from `lifestage_taxon_intersections.csv`, joined to the resolved baseline source key and derivation metadata. Do not collapse to one row per lifestage for adjudication. The 1000+ rows should be generated by rule and used as evidence; only rows in the action report require human review.

## Plan Shape

Use two waves:

1. Wave 1 creates the semantic adjudication artifact and a repeatable build script. It should not change runtime resolver ranking.
2. Wave 2 turns the artifact into enforcement by extending `dev/lifestage/validate_36.2.R` and `tests/testthat/test-eco_lifestage_data.R`.

This wave split matches the risk: first make the judgment surface visible, then make CI enforce it.

## Validation Architecture

Primary validation commands:

- `Rscript dev/lifestage/build_semantic_adjudication.R`
- `Rscript dev/lifestage/validate_36.2.R`
- `Rscript -e "devtools::test(filter='eco_lifestage_data')"`

Acceptance checks:

- Every resolved baseline `org_lifestage` has at least one adjudication row.
- Every resolved `org_lifestage + eco_group + kingdom + class_name` row in `lifestage_taxon_intersections.csv` has exactly one adjudication row.
- Every adjudication row has a valid status and reason.
- Every `semantic_change`, `ambiguous_route`, or `no_source_backed_candidate` retained curated row is represented in the adjudication artifact.
- No source-backed replacement that changes `harmonized_life_stage` or `reproductive_stage` is approved without an `approved_exception` row.
- Semantic-change examples such as Sapling are represented as ordinary adjudication rows, not special-case policy locks.
- `validate_36.2.R` fails if the adjudication artifact is missing, stale against baseline/derivation, or contains `needs_manual_review` for resolved baseline rows.
- `validate_36.2.R` fails if the adjudication artifact drops non-dominant species-group contexts.

## Risks And Pitfalls

- Do not treat ontology evidence as the final harmonized category. It selects candidate source concepts; it does not erase ECOTOX context.
- Do not globalize broad PO or UBERON concepts without checking each ECOTOX term and route context.
- Do not delete curated exceptions simply because a source-backed candidate exists.
- Do not make reports only in `dev/lifestage/` if downstream validation depends on them. Stable gates need stable package data under `inst/extdata/ecotox/`.
- Do not add live API calls to validation. Phase 36.2 gates must be artifact-backed and deterministic.
