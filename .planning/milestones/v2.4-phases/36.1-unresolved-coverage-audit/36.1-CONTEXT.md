# Phase 36.1: Unresolved Coverage Audit - Context

**Gathered:** 2026-04-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Classify every unresolved lifestage term in `lifestage_baseline.csv` into a triage bucket, prototype 4 external data sources to determine which can programmatically resolve alias/synonym terms, and produce a committed audit artifact (`lifestage_audit.csv`) that end users can read and edit. Ship gate: no unresolved term may remain unclassified.

Out of scope: writing permanent adapter code (that's Phase 36.2), expanding the derivation CSV beyond administrative terms, changing the rebuild path logic beyond adding the `cli_warn` gate.

</domain>

<decisions>
## Implementation Decisions

### Triage classification scheme
- **D-01:** Every unresolved term gets one of 5 buckets: `alias_synonym`, `needs_ontology_expansion`, `ambiguous`, `administrative_noise`, `out_of_scope_biology`.
- **D-02:** The 4 administrative terms (Not coded, Not reported, Not intact, Unspecified) map to `administrative_noise` bucket with resolution path = manual derivation to `harmonized_life_stage = "Unspecified"`, `reproductive_stage = FALSE`.
- **D-03:** 100% classification required — every one of the 69 unresolved terms must have a bucket and a documented resolution path before the phase ships.

### Programmatic resolution strategy
- **D-04:** Prototype all 4 data sources via dev scripts to validate viability before committing to permanent adapters:
  1. **Broader OLS4 search** — relaxed parameters, synonym fields, cross-reference search
  2. **BioPortal Annotator** — REST API across 900+ ontologies (FBdv, ZFA, NCBI organismal)
  3. **Wikidata SPARQL** — lifestage concepts with exact_match/close_match to UBERON/PO
  4. **FAO/AGROVOC** — agricultural vocabulary for crop stages and aquaculture terms
- **D-05:** Test each source with both simple terms (Fry, Fingerling) and complex terms (Copepodid, Glochidia) to gauge coverage breadth.
- **D-06:** Prototype results determine which sources become permanent adapters in Phase 36.2.

### Audit artifact
- **D-07:** Committed as `inst/extdata/ecotox/lifestage_audit.csv` — installed with the package, accessible via `system.file()`, editable by end users.
- **D-08:** Columns: `org_lifestage`, `triage_bucket`, `resolution_path`, `candidate_source`, `notes`.
- **D-09:** End users who force a rebuild can see this file to understand why a term is unresolved, and can edit their local copy to reclassify terms for their needs.

### Runtime gate
- **D-10:** The rebuild path (`.eco_patch_lifestage()`) emits `cli_warn` listing any terms not found in the audit CSV. Rebuild continues — unclassified terms land in `lifestage_review` as unresolved. User can then edit local audit CSV and re-run.
- **D-11:** No `cli_abort` for unclassified terms at runtime — warn but don't block.

### Dev/CI gate
- **D-12:** testthat test for CI enforcement: anti-join baseline unresolved rows against audit CSV. Any unclassified term fails CI. Catches regressions before publish.

### Prototyping approach
- **D-13:** Dev scripts at `dev/lifestage/probe_*.R` (e.g., `probe_ols4.R`, `probe_bioportal.R`, `probe_wikidata.R`, `probe_agrovoc.R`). Quick experiments, disposable after results inform Phase 36.2.
- **D-14:** Each probe script tests a representative sample of terms from each triage bucket and records what the source returned — used to populate `candidate_source` column in the audit CSV.

### Claude's Discretion
- Exact probe script structure and test term selection
- Order of source prototyping
- Whether to combine probe results into a single summary or keep per-source
- Validation script naming and structure (validate_36.1.R)
- Exact `notes` column content in audit CSV

</decisions>

<specifics>
## Specific Ideas

- Probe scripts should test both simple aliases (Fry, Fingerling — likely UBERON synonyms) and hard cases (Copepodid, Trophozoite — may need specialist ontologies) to calibrate expectations per source.
- The `resolution_path` column in the audit CSV should name the specific source that can resolve the term (e.g., "ols4_synonym", "bioportal_zfa", "wikidata", "agrovoc", "manual_curator", "permanently_unresolved") — this directly feeds Phase 36.2 adapter prioritization.
- Administrative terms (4 total) should get derivation rows in this phase since the mapping is trivial and unambiguous (all → "Unspecified").

</specifics>
