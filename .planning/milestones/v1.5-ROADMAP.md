# Roadmap: ComptoxR v1.5 Swagger 2.0 Body Schema Support

**Created:** 2026-01-29
**Milestone:** v1.5
**Requirements:** 19 total
**Phases:** 7-9 (continues from v1.4 Phase 6)

## Overview

This milestone extends the stub generation pipeline to support Swagger 2.0 schemas (AMOS, RDKit, Mordred) alongside existing OpenAPI 3.0 support. The parser will detect schema versions, extract body parameters from Swagger 2.0's `parameters[].in="body"` structure, resolve references to `#/definitions/`, and regenerate affected stubs with correct body handling.

## Phases

### Phase 7: Version Detection and Body Extraction

**Goal:** Parser correctly identifies schema versions and extracts body parameters from both Swagger 2.0 and OpenAPI 3.0 formats.

**Dependencies:** None (foundation phase)

**Plans:** 2 plans

Plans:
- [x] 07-01-PLAN.md — Add preflight filter, version detection, and Swagger 2.0 body extraction functions
- [x] 07-02-PLAN.md — Wire version detection into openapi_to_spec() parser pipeline

**Requirements:**
- VERS-01: Parser detects Swagger 2.0 schemas via `swagger` field at root
- VERS-02: Parser detects OpenAPI 3.0 schemas via `openapi` field at root
- VERS-03: Version detection routes to appropriate extraction logic
- BODY-01: Extract body schema from `parameters[].in="body"` for Swagger 2.0
- BODY-02: Extract body schema from `requestBody.content` for OpenAPI 3.0 (existing)
- BODY-03: Resolve `$ref` to `#/definitions/{SchemaName}` for Swagger 2.0
- BODY-04: Handle inline object schemas (no $ref, direct properties)
- BODY-05: Validate single body parameter constraint per Swagger 2.0 spec
- BODY-06: Validate body/formData mutual exclusivity per spec
- FILT-01: Add "preflight" to ENDPOINT_PATTERNS_TO_EXCLUDE regex

**Success Criteria:**
1. Parser detects `swagger: "2.0"` and routes to Swagger 2.0 extraction logic
2. Parser detects `openapi: "3.x.x"` and routes to OpenAPI 3.0 extraction logic
3. Swagger 2.0 POST endpoints with `parameters[].in="body"` extract body schema correctly
4. Swagger 2.0 endpoints with inline schemas (no $ref) extract properties directly
5. Parser validates Swagger 2.0 spec constraints (single body param, no body+formData mixing)
6. Preflight endpoints are filtered out during spec generation

**Key Files:**
- `dev/endpoint_eval/00_config.R` — Add "preflight" to ENDPOINT_PATTERNS_TO_EXCLUDE
- `dev/endpoint_eval/01_schema_resolution.R` — Add version detection, extract_swagger2_body_schema(), update extract_body_properties()
- `dev/endpoint_eval/04_openapi_parser.R` — Wire version detection into openapi_to_spec()

### Phase 8: Reference Resolution

**Goal:** Parser resolves schema references across both Swagger 2.0 (`#/definitions/`) and OpenAPI 3.0 (`#/components/schemas/`) with nested reference support.

**Dependencies:** Phase 7 (requires version detection and body extraction)

**Plans:** 2 plans

Plans:
- [x] 08-01-PLAN.md — Enhance resolve_schema_ref() with version-aware fallback chain and cli error handling
- [x] 08-02-PLAN.md — Wire version context through extraction functions and verify end-to-end

**Requirements:**
- REF-01: Fallback chain checks `#/definitions/` then `#/components/schemas/`
- REF-02: Version context passed through entire resolution chain
- REF-03: Nested `$ref` resolution with configurable depth limit (default: 3)

**Success Criteria:**
1. Parser resolves Swagger 2.0 references like `$ref: "#/definitions/BatchSearch"` to correct schema
2. Parser resolves OpenAPI 3.0 references like `$ref: "#/components/schemas/ChemicalList"` to correct schema
3. Parser uses fallback chain (checks both locations) when version is ambiguous
4. Nested references resolve correctly up to configured depth limit (default: 3 levels)
5. Version context flows through all resolution function calls

**Key Files:**
- `dev/endpoint_eval/01_schema_resolution.R` — Update resolve_schema_ref() with fallback chain, version parameter, recursion depth tracking

### Phase 9: Integration and Validation

**Goal:** All Swagger 2.0 POST endpoints generate correct stubs, OpenAPI 3.0 stubs remain unchanged, and affected microservice stubs are regenerated.

**Dependencies:** Phase 8 (requires reference resolution)

**Plans:** 1 plan

Plans:
- [x] 09-01-PLAN.md — Validate integration with baseline comparison, regenerate stubs, verify no regressions

**Requirements:**
- INTEG-01: Version detection wired into `openapi_to_spec()` main parser
- INTEG-02: All Swagger 2.0 POST endpoints with body schemas generate stubs
- INTEG-03: OpenAPI 3.0 stubs unchanged (no regression)
- INTEG-04: Regenerate AMOS stubs with correct body parameters
- INTEG-05: Regenerate RDKit stubs with correct body parameters
- INTEG-06: Regenerate Mordred stubs with correct body parameters

**Success Criteria:**
1. AMOS schema POST endpoints generate function stubs with correct body parameters
2. RDKit schema POST endpoints generate function stubs with correct body parameters
3. Mordred schema POST endpoints generate function stubs with correct body parameters
4. All generated Swagger 2.0 stubs include roxygen documentation for body parameters
5. OpenAPI 3.0 stubs (CompTox Dashboard) are identical before and after changes
6. Empty POST detection still works correctly for both schema versions

**Key Files:**
- `dev/endpoint_eval/04_openapi_parser.R` — Ensure version detection integrated throughout
- `dev/endpoint_eval/07_stub_generation.R` — Verify empty POST detection handles both versions
- `R/chemi_amos_*.R` — Regenerated AMOS stubs
- `R/chemi_rdkit_*.R` — Regenerated RDKit stubs
- `R/chemi_mordred_*.R` — Regenerated Mordred stubs
