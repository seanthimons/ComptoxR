# Milestone v1.4: Empty POST Endpoint Detection

**Status:** SHIPPED 2026-01-29
**Phases:** 6
**Total Plans:** 1

## Overview

This milestone adds detection and reporting for incomplete POST endpoints in the stub generation pipeline. When a POST endpoint has no query parameters, no path parameters, and an empty body schema, it cannot accept meaningful input and should be skipped with clear user notification.

## Phases

### Phase 6: Empty POST Detection

**Goal**: Users receive clear feedback when POST endpoints are skipped due to having no parameters or body properties
**Depends on**: Phase 5 (v1.3 complete)
**Requirements**: DETECT-01, DETECT-02, DETECT-03, DETECT-04, NOTIFY-01, NOTIFY-02, NOTIFY-03, SCOPE-01, SCOPE-02
**Success Criteria** (what must be TRUE):
  1. POST endpoints with no query params, no path params, and empty body schema are skipped during generation
  2. User sees `cli::cli_warn()` message for each skipped endpoint during generation run
  3. User sees summary of all skipped endpoints at end of generation run
  4. GET endpoints with no parameters still generate correctly (not affected)
  5. POST endpoints with ANY parameter (query, path, or body with properties) still generate correctly (not affected)

## Milestone Summary

**Decimal Phases:** None

**Key Decisions:**
- Detection runs after ensure_cols() but before stub generation
- Skipped/suspicious tracked in package-level .StubGenEnv environment
- Log files written to dev/logs/ with timestamp suffix
- Suspicious = query params only with empty body (heuristic)

**Issues Resolved:**
- POST endpoints with incomplete schemas were generating non-functional stubs
- Users had no visibility into which endpoints were skipped or why
