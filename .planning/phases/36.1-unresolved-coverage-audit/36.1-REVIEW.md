## Warnings

### WR-01: `!nrow(accepted)` used as boolean — falsy-zero anti-pattern

**File:** `R/eco_lifestage_patch.R:691`
**Issue:** The condition `if (!nrow(accepted))` tests whether the data frame has zero rows by
negating the integer row count. In R, `!0L` is `TRUE` and `!1L` is `FALSE`, so this works
today. However it is a well-known bug magnet: if `accepted` were ever `NULL` (e.g., because
`dplyr::filter` returns NULL in an edge case after a type change) `nrow(NULL)` returns `NULL`
and `!NULL` throws an error. The idiomatic guard is `nrow(...) == 0L`.

**Fix:**
```r
# Current (line 691)
if (!nrow(accepted)) {

# Corrected
if (nrow(accepted) == 0L) {
```

### WR-02: `baseline_matches` scalar/vector mismatch with `identical()`

**File:** `R/eco_lifestage_patch.R:303`
**Issue:** `baseline_releases` is computed as `unique(stats::na.omit(baseline$ecotox_release))`
and could be a character vector of length > 1 if the baseline CSV contains rows from multiple
releases. `identical(baseline_releases, ecotox_release)` then compares a vector to a scalar,
returning `FALSE` silently — which causes the baseline to be treated as non-matching and falls
through to live resolution, potentially bypassing the committed baseline entirely without any
warning to the user.

The intent is clearly to assert a single-release baseline. The check should first enforce
length-1 before the `identical()` comparison, or use `length(baseline_releases) == 1 &&
identical(baseline_releases, ecotox_release)`.

**Fix:**
```r
# Current (lines 302-303)
baseline_releases <- unique(stats::na.omit(baseline$ecotox_release))
baseline_matches <- identical(baseline_releases, ecotox_release)

# Corrected — fail loudly if baseline has mixed releases, check scalar equality
baseline_releases <- unique(stats::na.omit(baseline$ecotox_release))
if (length(baseline_releases) > 1) {
  cli::cli_abort(c(
    "Committed lifestage baseline contains rows from multiple releases.",
    "x" = "Releases found: {.val {paste(baseline_releases, collapse = ', ')}}",
    "i" = "Re-generate the baseline for a single ECOTOX release."
  ))
}
baseline_matches <- length(baseline_releases) == 1 && identical(baseline_releases, ecotox_release)
```

### WR-03: Audit gate uses `readr::read_csv` without a dev-path fallback

**File:** `R/eco_lifestage_patch.R:821-841`
**Issue:** `.eco_lifestage_baseline_path()` and `.eco_lifestage_derivation_path()` both probe
`system.file(...)` first and then fall back to `file.path("inst", "extdata", ...)` for
in-development use. The audit gate at lines 821-841 only calls `system.file(...)` and has no
dev-path fallback. When the package is loaded via `devtools::load_all()` during development,
`system.file()` returns an empty string and `file.exists("")` is `FALSE`, so the entire audit
gate is silently skipped — new unclassified terms added during development will never trigger
the `cli_warn`.

**Fix:**
```r
# After system.file() call, add dev fallback consistent with other path helpers:
audit_path <- system.file(
  "extdata", "ecotox", "lifestage_audit.csv", package = "ComptoxR"
)
if (!nzchar(audit_path)) {
  audit_path <- file.path("inst", "extdata", "ecotox", "lifestage_audit.csv")
}
if (nzchar(audit_path) && file.exists(audit_path)) {
  # ... existing gate logic
}
```

### WR-04: `strict_seed` flag may incorrectly evaluate to `TRUE` in auto mode

**File:** `R/eco_lifestage_patch.R:795`
**Issue:** The `strict_seed` flag is computed as:

```r
strict_seed <- seed$cache_source %in% c("cache", "baseline") && refresh %in% c("cache", "baseline")
```

Both conditions must be true. The intent is to abort when the user explicitly asked for cache or
baseline mode and it is incomplete. However when `refresh = "auto"` resolves to a cache or
baseline source (i.e., `seed$cache_source` is `"cache"` or `"baseline"`), `strict_seed` is
still `FALSE` because `refresh = "auto"` fails the second condition — this is correct.

The edge case is when `refresh = "cache"` or `"baseline"` but `force = TRUE` internally
re-invokes `.eco_lifestage_load_seed_cache` with `refresh = "auto"`. That recursive call
returns with `refresh_mode = "auto"` but the outer `refresh` variable still holds `"cache"` or
`"baseline"`. So `strict_seed` will be `TRUE` for the outer call even after the forced
fallback, and any missing terms will still trigger `cli_abort` instead of live resolution.

The `force` parameter is supposed to permit fallback to live, but `strict_seed` never consults
`force`. The guard at line 803 should also check `!isTRUE(force)`.

**Fix:**
```r
# Current (line 795)
strict_seed <- seed$cache_source %in% c("cache", "baseline") && refresh %in% c("cache", "baseline")

# Corrected — respect force fallback
strict_seed <- seed$cache_source %in% c("cache", "baseline") &&
  refresh %in% c("cache", "baseline") &&
  !isTRUE(force)
```

### IN-01: `lifestage_derivation.csv` has four duplicate `NA,NA` administrative rows

**File:** `inst/extdata/ecotox/lifestage_derivation.csv:55-58`
**Issue:** The four administrative-noise rows at the bottom of the derivation CSV all have
identical content (`NA,NA,Unspecified,FALSE,administrative_noise_audit`). The consuming code in
`.eco_lifestage_derivation_map()` calls `dplyr::distinct(.data$source_ontology,
.data$source_term_id, .keep_all = TRUE)` which will collapse all four rows to a single row.

This is harmless in terms of runtime behavior because the deduplication runs before any join.
However, it creates confusion for curators reading the raw CSV (four seemingly identical rows
with no distinguishing key) and the validate script at line 96 hard-codes `>= 4L` as a lower
bound to assert all four rows exist — the number of rows is an implementation detail, not a
semantic contract.

**Suggestion:** Either add a `org_lifestage` key column to these rows (`Not coded`, `Not
intact`, `Not reported`, `Unspecified`) so they are individually identifiable, or reduce to one
canonical `NA,NA` row and update the validate script's lower-bound assertion to `>= 1L`.

### IN-02: `probe_wikidata.R` embeds a personal email address in a committed file

**File:** `dev/lifestage/probe_wikidata.R:57`
**Issue:** The Wikidata SPARQL `User-Agent` header includes a hardcoded personal email address:

```r
`User-Agent` = "ComptoxR/dev probe script (verfassergeist@gmail.com)"
```

This is a dev-only probe script, but it is committed to the repository. Email addresses in
source code are harvested by scrapers and can cause increased spam load on the individual.

**Suggestion:** Replace with a generic contact (e.g., `ccte_api@epa.gov` which is already the
established CompTox contact address, or a GitHub issues URL):

```r
`User-Agent` = "ComptoxR/dev probe script (https://github.com/EPA-CCTE/ComptoxR/issues)"
```

### IN-03: `validate_36.1.R` uses `stopifnot()` for user-facing validation — poor diagnostics

**File:** `dev/lifestage/validate_36.1.R:23,33,52,63,96,114,117`
**Issue:** The validation script uses `stopifnot()` throughout for condition checks. When an
assertion fails during a curator run, `stopifnot()` produces a terse R error message with a
code expression rather than a human-readable diagnosis. The surrounding code already uses `cli`
for all output, so a `cli::cli_abort()` with an explanatory message would be more consistent
and actionable for curators.

The check at line 116 (`no_abort_in_gate <- !any(grepl(...))`) is also misleadingly named:
the variable name implies there should be no `cli_abort` in the audit gate block, but the grep
pattern `"cli_abort.*audit"` tests for co-occurrence of both strings on the same line — it
would fail to catch a multi-line `cli_abort(...)` call that references `audit` on a separate
line. This is low risk but the pattern is fragile.

**Suggestion:** Replace `stopifnot()` with `cli::cli_abort()` for assertions that face
curators. Tighten the line 116 grep to match the actual gate string (as line 113 does).

### IN-04: All four probe scripts call `devtools::load_all()` at the top level

**File:** `dev/lifestage/probe_agrovoc.R:6`, `dev/lifestage/probe_bioportal.R:7`,
`dev/lifestage/probe_ols4.R:7`, `dev/lifestage/probe_wikidata.R:7`
**Issue:** Each probe script runs `devtools::load_all(".", quiet = TRUE)` unconditionally. This
requires `devtools` to be installed and the working directory to be the package root. If a
curator runs the script from a different directory (e.g., `Rscript dev/lifestage/probe_ols4.R`
from a parent directory), `load_all()` will silently load the wrong package or fail with a
cryptic error.

The probe scripts do not appear to use any internal package functions — they only call `httr2`,
`jsonlite`, `cli`, `purrr`, and `tibble` directly. If they genuinely do not need internal
functions, the `load_all()` call can be removed and replaced with explicit `library()` calls for
each dependency. If internal functions are needed, a working-directory guard should be added.

**Suggestion (if no internal functions needed):**
```r
# Replace devtools::load_all() with:
library(httr2)
library(jsonlite)
library(cli)
library(tibble)
library(purrr)
```

Or add a guard:
```r
if (!file.exists("DESCRIPTION")) stop("Run from package root directory.")
suppressPackageStartupMessages(devtools::load_all(".", quiet = TRUE))
```
